## 2024-05-18 - Prevent Unnecessary Settings UI Re-renders
**Learning:** In this architecture, `App.tsx` handles low-level state like hue streaming health polling (`activeOutputTargets`) and tray interactions, which frequently update its internal state without necessarily changing user-facing settings props. `SettingsLayout`, despite being the large main UI container, does not default to `React.memo()`. Because `App.tsx` binds several props to `SettingsLayout` that are either stable (using `useCallback`) or only change during actual settings navigation, wrapping `SettingsLayout` in `React.memo` effectively blocks massive React re-render cascades when background polling tasks in `App.tsx` trigger state updates.
**Action:** When working on large React UI containers mounted inside orchestrator components (like `App.tsx`), always check if the orchestrator manages frequent state updates (e.g., polling). If so, ensure the layout container is memoized and verify that its callback props are wrapped in `useCallback` to maximize the impact of the memoization.
